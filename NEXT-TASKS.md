## 🎯 المهمة 1 — نظام تجهيز الطلبات (Fulfillment) الاحترافي — أولوية عالية

**الهدف:** الموظف المسؤول عن التجهيز يشوف **بشكل يومي** الطلبات اللي لازم يجهّزها، يجهّزها بصفر أخطاء، ويأشّر/يرحّل بسهولة — متل أنظمة شركات الشحن الكبيرة.

**الوضع الحالي (موجود):**
- دور `order_role='fulfillment'` بـ`profiles` + عرض «📦 طلبات التجهيز» (`OrdersScreen.jsx:2247,2681`) + عدّاد «X طلب يحتاج عملك» + بانر «وارد ينتظر التجهيز» + صلاحية تغيير الحالة (`canAdvanceOrders`).
- طابور بالحالات: pending(وارد)→preparing(تجهيز)→ready(جاهز) ثم مسار الشحن/الموتور (`orderStatus.js`).

**المطلوب (NEEDS):**
1. **فصل per-market + per-warehouse:** المجهّز يشوف **فقط** طلبات سوقه/مخزنه. تركيا = Fatima Ayoub؛ سوريا = شخص آخر بمخزن مختلف. (الفلترة بـ`order_market` + إضافة فصل المخزن.)
2. **شاشة تجهيز يومية مخصّصة:** «طلبات اليوم» + «متأخّرات الأمس» (طلبات أمس لسا ما جُهّزت/ما وصلت) — يعرف كم طلب اليوم، كم بقي، وأي طلب متأخّر يتابعه.
3. **فرز حسب جهة الإرسال:** مجموعات — موتور / شركة شحن (كل شركة) / استلام من المركز — حتى يعرف شو بدّو يجهّز ولمين.
4. **تأشير/ترحيل سهل (صفر أخطاء):** زر واضح لكل طلب «تم التجهيز» (ينقل pending→ready/preparing→الحالة التالية) + **إجراء جماعي (batch)** لتأشير عدة طلبات دفعة وحدة، وتأكيد بصري.
5. **ميزات تسهّل (اقتراحات):** طباعة قائمة تجهيز/ملصقات، عدّاد إنجاز اليوم، تنبيه بالطلبات العالقة، بحث/فلتر سريع، منع تخطّي خطوة.

**أسئلة لازم تُحسم قبل التنفيذ:**
- مين مجهّز سوريا (الاسم) وأي مخزن؟ (لضبط `order_role`/`order_market`).
- تأكيد ضبط Fatima Ayoub كـ`order_role=fulfillment`, `order_market=turkey`.
- هل نضيف عمود/جدول مخازن لربط المجهّز بمخزنه؟

## 🎯 المهمة 2 — تسليمات الشهر: ربح/خسارة قبل الأرشفة — أولوية عالية

**الهدف:** بآخر الشهر المالك يعرف: **كم طلب مسلّم · كم قطعة مباعة · التكلفة · الدخل · الربح أو الخسارة** — ثم يرحّل الطلبات للأرشيف يدوياً (مرة/الشهر، أحياناً متأخّر) بعد ما ياخد الحساب.

**الوضع الحالي (موجود):**
- تبويب «📦 تسليمات الشهر» (`OrdersScreen.jsx:533-755`، MonthlyDeliveriesTab) للمدير: فترة (شهر/مخصّص) + عدد المسلّم + الإيراد بالـUSD (سعر الكتالوج، مناعة ضد العروض عبر `orderUsd()`) + إجماليات TRY/SYP + عمولة + ملخّص منتجات + تصدير Excel + أرشفة يدوية (`handleMonthArchive` → `archived=true`).
- تكاليف موجودة بـ`product_economics` (`cost_usd, ad_cost_usd, shipping_cost_usd`) لكنها تُعرض فقط بـ`ProfitabilityScreen` — **مش بتسليمات الشهر**.

**المطلوب (NEEDS):**
1. **إضافة الربح/الخسارة لتبويب تسليمات الشهر:** لكل فترة احسب واعرض:
   - عدد القطع المباعة (Σ الكميات).
   - الإيراد (موجود) − **التكلفة** (Σ الكمية × `cost_usd` + الشحن + الإعلانات؟) = **صافي ربح/خسارة** بالـUSD.
2. **تقرير شهري واضح** قبل الأرشفة (P&L مبسّط) + إضافته لتصدير Excel.
3. (اختياري) فصل P&L تركيا مقابل سوريا.
4. إبقاء **الترحيل اليدوي للأرشيف** كما هو (المالك يرحّل لما يجهز الحساب).

**أسئلة لازم تُحسم:**
- التكلفة = تكلفة المنتج فقط، وللا + الشحن + الإعلانات؟ (أي حقول من `product_economics` ندخل بالربح).
- هل `product_economics.cost_usd` معبّأ لكل المنتجات؟ (لو لأ، نحتاج إدخالها — قد يكون نقطة بيانات ناقصة).
- مصدر تكلفة الشحن/الإعلانات per-order أو إجمالي شهري؟

## 🎯 المهمة 3 — أسئلة التدريب أذكى وأوسع (Claude API) — أولوية متوسطة

**الهدف:** أسئلة تدريب ذكية، متنوّعة، مفيدة — مربوطة بـClaude API (متل لوزي)، والجواب الصح مش دايماً أول خيار.

**الوضع الحالي (مفاجأة — موجود جزئياً):**
- الأسئلة **فعلاً تُولَّد بـClaude** (`generate-quiz` edge fn، `claude-sonnet-4-6`، + web search لمكوّنات/أسئلة العملاء) وتُخزَّن بـ`quiz_questions` (`source='ai'`).
- الأجوبة **مخلوطة فعلاً** (`shuffleOptions` بـ`TrainingScreen.jsx:51-77`) — الصح **مش** دايماً أول خيار.
- شهادة عند ≥80% + تتبّع إجابات.

**التشخيص المرجّح للشكوى:** التوليد الذكي **مش شغّال فعلياً** → التطبيق يقع على الأسئلة **الثابتة (static)** الضعيفة (محدودة، وقد يكون الصح فيها أول خيار). السبب المحتمل: `ANTHROPIC_API_KEY` غير مضبوط، أو ما في جدولة يومية للتوليد، أو الكتالوج بالبرومبت قديم.

**المطلوب (NEEDS):**
1. **تأكيد/تفعيل التوليد الذكي:** تحقّق `ANTHROPIC_API_KEY` مضبوط + `generate-quiz` منشورة + **جدولة يومية** (pg_cron) تولّد أسئلة كل يوم لكل تصنيف.
2. **توسيع وتعميق الأسئلة:** زيادة التصنيفات/العدد اليومي، ربط البرومبت بجدول `products` الحقيقي (مش كتالوج hardcoded)، تنويع الصعوبة.
3. **تقليل/تحسين الأسئلة الثابتة الضعيفة** (أو إلغاء fallback اللي الصح فيه أول خيار).
4. (اختياري) ربط بلوزي/تخصيص حسب أداء المتعلّم.

**أسئلة لازم تُحسم:**
- هل `ANTHROPIC_API_KEY` مضبوط على Supabase حالياً؟ (يحدّد إذا التوليد شغّال أصلاً).
- كم سؤال/يوم وأي تصنيفات يريد المالك؟
