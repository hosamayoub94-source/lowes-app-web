# تصميم: إعادة هيكلة الدخول والأدوار — LOWE'S

> التاريخ: 2026-06-08 · الحالة: معتمد من المالك · المشروع: lowes-app-web

## السياق والمشكلة
لوحة الدخول الحالية تعرض **قائمة بأسماء كل الموظفين** حسب الدور (`LoginScreen` → `listActiveProfiles({roleType})`) — تسريب أمني: أي شخص يفتح الرابط يكشف كل أسماء الفريق والمدراء. كما أنّ إضافة أدوار التوزيع (field_rep/marketer/...) جعلت لوحة الدخول تعرض 11 خانة. المالك يريد: دخول آمن بلا أسماء، فصل قناة المسوّقين/الموزّعين، وتنظيم احترافي للأدوار.

## القرارات المعتمدة
1. **مصادقة بكود شخصي + PIN** (لا قوائم أسماء).
2. **لوحة دخول بخانتين:** «فريق LOWE'S» · «شبكة المبيعات».
3. **صفحة الفريق** (داخل التطبيق) للموظفين فقط — لا تظهر لشبكة المبيعات.
4. **٣ أدمن** (حسام/ريم/أماني) يمنحون الأدوار والصلاحيات للباقين عبر شاشة المستخدمين الموجودة.

## القسم ١ — الدخول والأمان
- **DB:** عمود `profiles.login_code TEXT UNIQUE` + `GRANT SELECT (login_code) ON profiles TO anon, authenticated` (لازم للتحقّق وقت الدخول). توليد كود لكل الموظفين الحاليين (مشتق من الاسم/مختصر + رقم، فريد). الأدمن يعدّله من «المستخدمون».
- **تصنيف القناة:** دالة تربط `role_type` بقناة: `field_rep/marketer/supervisor/supervisor_manager/area_agent/store → 'sales'`، الباقي `→ 'team'`. عمود محسوب منطقياً (في الكود) لا في DB.
- **authService:** `signInWithCode(login_code, pin, channel)` — يبحث بالكود، يتحقّق من PIN (نفس مسار verify-pin الحالي عبر اسم البروفايل المُكتشف من الكود)، ويرفض إذا قناة المستخدم ≠ القناة المختارة.
- **LoginScreen:** خطوتان — (1) خانتان (فريق/مبيعات)، (2) حقل كود + PinPad. حذف خطوة «اختيار الاسم» نهائياً.
- **توافقية:** نولّد الأكواد لكل الـ56 قبل النشر؛ الأدمن يوزّعها. (لا fallback بالاسم — حسماً للتسريب.)

## القسم ٢ — صفحة الفريق (Team/Org)
شاشة `/team` يُعاد بناؤها بتابات (تظهر فقط لقناة team):
1. **Management** — org-chart بمسميات إنجليزية، كل المناصب حتى الشاغرة (Vacant): CEO & Founder · COO · General Manager–Syria · Sales Manager · Marketing Manager · Media Buyer · Finance/Accountant · Warehouse Manager · Social Media Manager. (قائمة المناصب ثابتة في `src/data/orgChart.js`، تُطابَق مع `profiles` لإظهار الشاغل أو Vacant.)
2. **الموظفون** — كل الموظفين أبجدياً (بمن فيهم حاملو الأدوار الإدارية).
3. **شبكة النجوم** — المسوّقات/المشرفات وفرقهم (شجرة — يعيد استخدام `my_downline`/خدمة mlm).
4. **بانرات:** الموزّعون · المتاجر أونلاين · الوكلاء — كل بانر يفتح قائمة تلك الفئة (فلترة `profiles` بـseller_type/role).
- **الوصول:** حارس على `/team` (وعناصر التنقّل) يمنع قنوات المبيعات. الراوتر: `roles` تستثني field_rep/marketer/...

## القسم ٣ — الصلاحيات
- يبقى نموذج `resolvePermissions` (role defaults + extra − denied). الأدمن الثلاثة يملكون الكل ويوزّعون.
- شبكة المبيعات: أدوارها الحالية (field_rep/marketer/...) صلاحياتها محدودة سلفاً (لا VIEW_NETWORK للأسماء العامة، لا وصول لـ/team).

## ملفات تتأثّر
- `supabase/`: migration جديد (login_code + توليد).
- `src/services/authService.js` (signInWithCode), `src/screens/LoginScreen.jsx` (إعادة بناء), `src/data/teams.js` أو ملف جديد (channelForRole), `src/screens/admin/AdminUsersScreen.jsx` (حقل/عرض login_code), `src/data/orgChart.js` (جديد), `src/screens/TeamScreen.jsx` (إعادة بناء بتابات), `src/routes/AppRoutes.jsx`+`navigation.js` (حارس /team).

## المخاطر
- **قفل الدخول:** كل موظف يحتاج كوده الجديد قبل النشر → نولّد ونعرضها للأدمن أولاً، ثم ننشر.
- **GRANT:** عمود login_code يُقرأ وقت الدخول → يحتاج GRANT صريح وإلا ينكسر الدخول.
- لا fallback بالاسم = حسم التسريب، بثمن أنّ من ينسى كوده يحتاج الأدمن.

## التحقّق
- توليد الأكواد للـ56 ثم محاولة دخول بكود+PIN لحساب اختبار من كل قناة.
- محاولة دخول بكود قناة «مبيعات» على خانة «فريق» → يُرفض.
- مستخدم شبكة مبيعات لا يرى `/team` ولا أي اسم.
