========== [4] HIGH (user-facing) — accounting ==========
TITLE: Non-atomic wallet transfer: leg 2 failure leaves an orphaned, unbalanced leg 1
LOC:   src/screens/AccountingScreen.jsx:202-230  [data-integrity]
DESC:  handleTransfer (wallet-to-wallet transfer) inserts the OUT leg and the IN leg as two separate sequential `await createEntry(...)` calls. There is no DB transaction and no compensating rollback. If the first insert succeeds but the second throws (network drop, RLS denial, validation error), the source wallet is permanently debited (TRANSFER_OUT persisted) with no matching credit. The treasury totals then silently lose money — the books no longer balance and there is no way to detect the orphan because the two legs are not even linked (see next finding). Unlike the service-level createTransfer (which uses a single insert([legOut, legIn])), this UI path does two round-trips.
FIX:   Mirror the existing atomic service pattern. Add a `createWalletTransfer` service function that builds both legs with a shared `transfer_group` (use globalThis.crypto.randomUUID() as in createTransfer at accountingService.js:118) and commits them in a single statement: `supabase.from('accounting_entries').insert([legOut, legIn]).select()`. Have handleTransfer call this one function instead of two sequential createEntry calls. This makes both rows commit-or-neither and links them for detectability. Note the wallet path supports cross-currency (amtFrom != amtTo) which the current createTransfer does not, so the new function must accept distinct out/in amounts and per-wallet payment_method/currency fields (use the existing walletAmounts helper logic). Minimum viable fallback if a new function is not added: in the catch block, if leg 1 returned a row, delete it (deleteEntry by id) before surfacing the error. Best long-term: a Postgres RPC so the pair is a true server-side transaction immune to client-side interruption between the two inserts.

========== [5] HIGH (user-facing) — warehouse-orders ==========
TITLE: Concurrent stock writes lose updates (no atomic decrement / lock)
LOC:   src/services/warehouseService.js:95-99, 225-239, 257-266, 296-305  [data-integrity]
DESC:  Every stock mutation is a read-then-write: _currentQty() reads quantity, then _setQty() upserts an absolute value. There is no DB-level lock, transaction, or atomic 'UPDATE ... SET quantity = quantity - x'. When two operations touch the same (warehouse, product) concurrently they both read the same starting qty and the second upsert overwrites the first, losing one decrement/increment. This is reachable in practice: advanceBatch fires syncOrderStock per order WITHOUT await (OrdersScreen.jsx:2648), so all orders in a batch that draw from the same sales warehouse deduct concurrently. Result: stock is under-deducted (sells phantom inventory) or over-restored.
FIX:   Suggested fix is correct. Implement Postgres RPCs (SECURITY DEFINER) that perform atomic UPDATE wh_stock SET quantity = quantity ± p_qty WHERE warehouse_id=... AND product_id=... with INSERT ... ON CONFLICT for the missing-row case, and have warehouseService.js call supabase.rpc(...) instead of _currentQty+_setQty for ALL mutators (receiveStock, allocateStock, adjustStock, _reserveOrder, _releaseOrder, reverseMovement). Mirror the existing pattern in src/modules/inventory/services/inventoryService.js (reserve_inventory/release_inventory). As a cheap interim mitigation also add `await` to the un-awaited syncOrderStock calls in OrdersScreen.jsx (lines 2648, 2595, 2791, 2807) to serialize the in-batch case, but this alone does not fix cross-user/device concurrency — the atomic RPC is required for full correctness.

========== [6] HIGH (user-facing) — warehouse-orders ==========
TITLE: syncOrderStock is fire-and-forget in batch advance, defeating the deliberate sequential design
LOC:   src/screens/OrdersScreen.jsx:2641-2659  [data-integrity]
DESC:  advanceBatch was made sequential on purpose (it awaits syncToSheet per order to avoid a sync storm), but syncOrderStock on line 2648 is NOT awaited. RETURNING`, or a wh_movements-driven recompute under a row lock). Awaiting at 2648 is the correct, in-scope fix for the reported batch bug.
