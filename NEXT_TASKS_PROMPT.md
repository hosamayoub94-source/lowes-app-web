## 🎯 المهمة: تست شامل بدور **المستخدم + الخبير**
خُذ دور **المستخدم** (أنشئ/عدّل/غيّر حالة/احذف/استرجع طلبات حقيقية بالموقع الحيّ + بالجدول) و**الخبير** (تحقّق من سلامة البيانات، طابق الأعداد، اصطد الحالات الحدّية والفجوات). الموقع المنشور: `lowes-app-web.vercel.app` (مسجّل دخول أدمن). فحص الجداول: عبر دالة Apps Script تشخيصية (gviz/جلب مُصادَق **محجوب**). استخدم حقن جلسة بالمتصفح للاستعلام/نداء edge (التوكن بـ`localStorage['sb-fghdumrgimoeqsafdhhh-auth-token']`؛ جدّده بالـrefresh_token إن انتهى).

### مصفوفة الاختبار (سوريا + تركيا · بالاتجاهين)
لكل سوق، اختبر **بطلب تجريبي مُعلَّم بوضوح** (احذفه بعد) كامل دورة الحياة وتحقّق فعلياً من **التطبيق + الجدول** بعد كل خطوة:
1. **إنشاء** طلب من التطبيق → يظهر بالجدول الصحيح (سوريا: ورقة LOWES Sales · تركيا: LOWES_TR/STRONG_TR حسب البراند) بالأعمدة الصحيحة (اسم/هاتف/مدينة/عنوان/منتجات/مبلغ/عملة/حالة).
2. **تعديل** بيانات العميل/الأصناف من التطبيق → ينعكس بالجدول (التطبيق مصدر الحقيقة لبيانات العميل/الأصناف).
3. **تغيير الحالة**:
   - من التطبيق → الجدول (تركيا: doPost يحدّث بالمكان · سوريا: doPost الحيّ **يتخطّى الموجود `duplicate:true`** — تحقّق هل الحالة تنعكس أم لا، ووثّق).
   - من الجدول → التطبيق (sheet-to-app عبر onSheetEdit). **سوريا: أكواد الفريق NEW/PACK/SHIP/DONE/CANC/Waiting/Settled لازم تُربَط للتطبيق** (TEAM_MAP — مُصلَح؛ تأكّد فعلياً).
4. **حذف ناعم** + **استرجاع** → **بلا تكرار، بلا عودة** (سوريا doPost duplicate:true يمنع — تحقّق حيّاً).
5. **طلب مُسلَّم** → انقله لتاب التسليمات يدوياً ثم أعد مزامنته → **يجب ألا يعود** (`terminal_no_readd` — مُصلَح؛ تأكّد).

### نقاط حرجة من جلسة 11 يونيو (تحقّق منها خصّيصاً)
- **سوريا — توحيد المسميات (مُصلَح):** الجدول يستخدم أكواد الفريق الإنجليزية، التطبيق العربي. الربط ثنائي عبر `TEAM_MAP` (sheet-to-app) + `SYRIA_TEAM_CODE` (sync-order-to-sheet). **اختبر: غيّر كود فريق بالجدول → التطبيق يطابق؛ والعكس.** طابق أعداد كل حالة بين الجدول والتطبيق.
- **تركيا — تتبّع يورتيتشي التلقائي (مبني):** مؤقّت Apps Script `pollYurticiStatuses` كل 10 دقائق (مشروع Dashboard 1ub2zm) + `track-yurtici` (المسار العام) عند فتح الشاشة. يستعلم API العام `yurticikargo.com/service/shipmentstracking?id=<رقم>&language=tr`. **اختبر: طلب بتركيا برقم تتبّع 116083… → حالته تتحدّث تلقائياً (مُسلَّم/في النقل).** شحنات **KP** (شركة أخرى) **تُتخطّى** (لا «pushed=46» ولا غارباج — كان بقّاً مُصلَحاً).
- **تركيا — عمود الحالة I = «تحذير» لا «رفض»** (مُصلَح بـrelaxAllStatusValidation): أي حالة (motor…) تُكتب بلا فشل مزامنة.
- **عدم التكرار/العودة:** فحص الجداول لأي Order ID مكرر (دالة findDuplicateOrderIds نمط [[live-apps-script-projects]]) أو طلب مُسلَّم عاد للتاب النشط.

### دور الخبير — افحص الفجوات
- **مصدر الحقيقة:** هل القاعدة (حالة سوريا app→sheet معطّلة بـduplicate:true؛ هل المالك يريدها تعمل؟) — وثّق القرار المطلوب.
- **اتساق الأعداد:** بطاقات التطبيق (وصل/شحن/تجهيز/وارد) مقابل الجداول. أي انحراف = فجوة.
- **حالات حدّية:** طلب بلا منتجات/بلا هاتف · عملات (SYP/USD/TRY) · دفع جزئي (عمودا K/O سوريا) · أسماء منتجات (الجرد يطابق الإنجليزي) · براند تركيا (LOWES_TR vs STRONG_TR).
- **المزامنة الفاشلة:** لوحة الأدمن `listFailedSync` — صفر فاشل؟ أعد المحاولة وأصلح السبب.
